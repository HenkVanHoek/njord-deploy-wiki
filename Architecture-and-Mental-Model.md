# Architecture & Mental Model 🧠

PiSelfhosting is built on clean, predictable, and modular design patterns. 
It leverages Docker Engine on Debian-based host operating systems, using 
locally compiled files as the Single Source of Truth (SST).

---

## 📂 Host Directory Structure

All deployed configurations and application data are strictly isolated 
under the following paths on your Raspberry Pi:

*   `/opt/piselfhosting_data/`  
    The main directory where persistent application configurations, databases, 
    and application data are stored.
*   `/opt/piselfhosting_data/output/`  
    The runtime target where the generated `docker-compose.yml`, `.env`, and 
    `deployment_context.json` are generated by the Configurator before 
    Ansible deployment.

---

## 🧪 The Variable & Macro System

PiSelfhosting supports fully portable templates. Variables are declared in 
each component's `template-config/variables.json` file. 

During compilation, Jinja2 evaluates these variables and injects them into the 
templates. We support two special macro bindings:

1.  `CONFIG_BASE_PATH`  
    Represents `/opt/piselfhosting_data` dynamically. Always use:  
    `{{ CONFIG_BASE_PATH }}/your-service/data`  
    This guarantees path portability across all deployment machines.
2.  `DOTENV`  
    Binds properties to the global `.env` file of the server. For example:  
    `{{ DOTENV.TZ }}` or `{{ DOTENV.DOMAIN }}`  
    This ensures that shared, security-critical, or network-wide settings 
    only need to be entered once.

## ⚙️ Backend Architecture (CQRS Pattern)

To separate read operations (queries) from write mutations (commands), our 
system is organized around a CQRS pattern:

```
+-------------------------------------------------------------+
|                     ComponentManager                        |
|                   (Orchestration Layer)                     |
+------------------------------+------------------------------+
                               |
               +---------------+---------------+
               |                               |
               v                               v
+------------------------------+ +----------------------------+
|       ComponentReader        | |      ComponentWriter       |
|         (Read Side)          | |        (Write Side)        |
+------------------------------+ +----------------------------+
```

*   **`ComponentReader`**: Handles all parsing and loading of component 
    metadata, templates, and variables.
*   **`ComponentWriter`**: Handles directory and skeletal creation, file saves, 
    deletions, and package definitions.
*   **`ComponentManager`**: The service layer that acts as the coordinator. 
    It runs validations, coordinates file generation, and feeds the deployment 
    engines.
